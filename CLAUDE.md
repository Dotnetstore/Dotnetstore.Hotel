This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is a hotel management application built on .NET Aspire. The repo currently contains only the Aspire scaffold (AppHost, ServiceDefaults, test templates) generated by `dotnet new aspire` — no domain/feature code exists yet. The sections below capture the intended architecture and conventions to follow as real projects and features are added.

## Commands

- Build: `dotnet build Dotnetstore.Hotel.slnx`
- Run the distributed app (via Aspire AppHost, starts the dashboard): `dotnet run --project src/Dotnetstore.Hotel.Shared.AppHost`
- Run all tests: `dotnet test Dotnetstore.Hotel.slnx`
- Run a single test project: `dotnet test tests/Dotnetstore.Hotel.Shared.AppHost.Tests/Dotnetstore.Hotel.Shared.AppHost.Tests.csproj`
- Run a single test by name: `dotnet test --filter "FullyQualifiedName~TestName"`

The solution file is `Dotnetstore.Hotel.slnx` (the new XML-based slnx format, not `.sln`).

## Current architecture

- **Target framework**: net10.0 across all projects, set centrally in [Directory.Build.props](Directory.Build.props) along with `Nullable`/`ImplicitUsings` enabled and shared assembly metadata.
- **`src/Dotnetstore.Hotel.Shared.AppHost`**: the Aspire orchestrator entry point ([AppHost.cs](src/Dotnetstore.Hotel.Shared.AppHost/AppHost.cs)). This is where resources (services, databases, caches, etc.) get registered with `DistributedApplication.CreateBuilder`. New services must be wired up here.
- **`src/Dotnetstore.Hotel.Shared.ServiceDefaults`**: shared cross-cutting extensions ([Extensions.cs](src/Dotnetstore.Hotel.Shared.ServiceDefaults/Extensions.cs)) referenced by every service project. `AddServiceDefaults()` wires up OpenTelemetry (tracing/metrics/logging), health checks (`/health`, `/alive`), and service discovery with resilient `HttpClient` defaults. Every new service project must call `builder.AddServiceDefaults()` and `app.MapDefaultEndpoints()`.
- **`tests/Dotnetstore.Hotel.Shared.AppHost.Tests`**: integration tests that boot the AppHost via `Aspire.Hosting.Testing` (`DistributedApplicationTestingBuilder`) and assert against running resources over HTTP. [IntegrationTest1.cs](tests/Dotnetstore.Hotel.Shared.AppHost.Tests/IntegrationTest1.cs) is a template stub showing the pattern.
- **`tests/Dotnetstore.Hotel.Shared.TestHelper`**: placeholder for shared test utilities (currently empty).
- `Directory.Build.props` adds `InternalsVisibleTo` for `$(MSBuildProjectName).Tests` and `Dotnetstore.Intranet.AppHost.Tests` on every project, so internals are test-visible without per-project `AssemblyInfo` edits.

## Planned solution structure

- **API/**: projects exposing HTTP APIs, built with minimal APIs.
- **Shared/**: shared libraries/utilities. One `SDK` project holds DTOs and an `IHttpClientFactory`-based client for consuming the API, so other projects don't duplicate contracts or HTTP plumbing.
- **UI/**: web/mobile front-end projects.

## Domain & application conventions

- **Vertical slices**: organize by feature, not by technical layer, so each feature can be developed/tested independently.
- **CQRS, hand-rolled**: separate command and query paths; do not pull in MediatR — write the dispatch plumbing ourselves.
- **DDD**: model the domain (rooms, bookings, customers, billing) to reflect real hotel-operation processes, not just CRUD entities.
- **Repository + Unit of Work**: encapsulate data access behind repositories; group multi-step operations into a unit of work for transactional consistency.
- **Service layer**: holds business logic/orchestration (validation, transactions) between API endpoints and the domain/data layer.
- **Options pattern** for configuration binding.
- **Dependency injection** throughout; avoid service locators/statics.

## Data

- **EF Core + PostgreSQL** as the primary data store. Design schemas with normalization and indexing in mind.

## API & auth

- **Minimal APIs** for endpoints.
- **JWT** for authentication; **role-based access control** for authorization. Users/roles live in the database (registration, password recovery, account management included).
- Standard secure-transport/secure-coding practices (HTTPS, input validation against SQL injection/XSS/CSRF).

## Observability

- OpenTelemetry for metrics/tracing (already wired via `ServiceDefaults`), structured logging, and distributed tracing across services.
- Audit logging for changes to critical data and user actions, separate from general application logs.

## Testing

- **xUnit** + **Shouldly** for assertions, **Moq** for mocking dependencies.
- **Testcontainers** for PostgreSQL-backed integration tests (real DB in an isolated, reproducible container rather than mocking the database).
- Aspire integration tests boot real resources via `Aspire.Hosting.Testing` (see `IntegrationTest1.cs` pattern above).
